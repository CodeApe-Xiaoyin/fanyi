# Fanyi · YouTube 智能双语字幕扩展 · 开发项目规划书

> 版本：v1.0 · 草案
> 编写日期：2026-04-27
> 阶段：自用 MVP → 商业化演进

---

## 0. 一句话定位

一款 Chrome 扩展，在观看 YouTube 时实时生成「**先理解、后翻译**」的高质量双语字幕；**用户自带 LLM API Key**，可对接任意 OpenAI 兼容 / Anthropic / Gemini / 自定义协议厂商；架构上预留商业化能力（云端代理、付费门控、用量计费），但 MVP 阶段完全本地化、零后端。

---

## 1. 目标与非目标

### 1.1 目标（自用 MVP，4–6 周内可日常使用）

- M1：YouTube 视频自动加载双语字幕（基于原生字幕 + LLM 智能断句翻译）
- M2：支持任意 OpenAI 兼容厂商（含本地 Ollama / vLLM / 第三方代理 / OpenRouter）以及 Anthropic、Gemini 官方协议
- M3：可自定义字幕样式（字体、颜色、背景、描边、位置）
- M4：侧边栏显示完整字幕剧本，点击任一句跳转到对应时间点
- M5：字幕结果按 (videoId × 模型 × 目标语言) 缓存，二次打开秒开
- M6：**逐词高亮跟随**（MVP 用字符等比估算，Phase 3 升级到 forced alignment）—— 视觉上对标雪球的下划线高亮
- M7：**双语排版默认即专业级**（不依赖用户调样式就能看着舒服） —— 见第 15 节

### 1.2 远期目标（商业化阶段，留接口）

- C1：服务器代理 LLM/ASR 调用（用户无需自己填 key），后端记录用量
- C2：原生字幕缺失时的 ASR 转录（Whisper / Deepgram / 自部署 faster-whisper）
- C3：逐词时间戳级别的高亮跟随（依赖 ASR 或对齐服务）
- C4：账户体系 + 订阅付费 + 团队/共享字幕库
- C5：iOS Safari / Edge / Firefox 多浏览器兼容
- C6：垂直场景增强（编程教程术语保留、TED 演讲学术风格、财经新闻数据精准）

### 1.3 非目标（明确不做）

- 不做视频下载 / 离线播放
- 不做 AI 配音 / 换音
- 不在 MVP 阶段做账户系统、付费、服务端
- 不与 YouTube TOS 抵触的功能（绕过广告、绕过年龄验证等）

## 2. 竞品对标速览

| 维度 | 雪球双语字幕（对标） | 本项目（差异化） |
|---|---|---|
| 商业模式 | 免费，靠雪球英语 App 补贴 AI 成本 | 自带 Key 免费 + 未来云代理订阅 |
| LLM 选择 | 后台固定，用户不可选 | 用户完全自由，多厂商可切换 |
| 翻译风格 | 固定 | 可选（信达雅 / 直白 / 术语保留） |
| 上下文增强 | 推测仅句内 | 注入视频标题、章节、频道作 system context |
| 隐私 | 视频 ID + 字幕过其服务器 | MVP 完全直连 LLM 厂商，零中间方 |
| 断句算法 | 滑窗拼句（推测） | LLM 句界检测 + （后期）VAD 双信号 |

**核心差异化**：质量上限随用户预算上限走，雪球的成本约束我们没有。

### 3.2 三层职责边界

| 层 | 职责 | 不允许做的事 |
|---|---|---|
| **接口层** | DOM 操作、消息收发、用户交互、表单校验、UI 状态管理 | 直接调用 fetch / 调用 LLM / 直接读 chrome.storage |
| **业务逻辑层** | 用例编排、断句/分配算法、Prompt 构建、批处理策略、缓存策略决策 | 引用任何 chrome.* / DOM / 具体 SDK / fetch |
| **基础设施层** | 实现 Port 接口、与外部世界 IO（HTTP、Storage、DOM 抓取） | 包含业务规则；上层逻辑不应该出现在这里 |

**强制依赖方向**：接口层 → 业务层 → Port 接口 ← 基础设施层
（基础设施层「实现」业务层定义的接口，业务层不知道具体实现是谁）

### 3.3 组合根（Composition Root）

在扩展启动入口（`background/service-worker.ts` 与 `content-script` 入口）通过一个 `Container` 完成依赖装配；切换实现只需改装配。

```ts
// src/composition/container.ts
export function buildContainer(env: 'self-hosted' | 'cloud'): Container {
  const storage = new ChromeStorageAdapter();
  const cache = new IndexedDBCacheAdapter();
  const llm = new ProviderRegistry(storage); // 根据用户配置动态选择
  const subtitleSrc = new YouTubeTimedtextFetcher();
  const auth = env === 'cloud' ? new RemoteAuthGate(...) : new LocalAuthGate();
  const telemetry = env === 'cloud' ? new RemoteTelemetry(...) : new NoopTelemetry();

  return {
    pipeline: new SubtitlePipelineUseCase({ llm, subtitleSrc, cache, auth, telemetry }),
    translate: new TranslateBatchUseCase({ llm, cache }),
    // ...
  };
}
```

切到商业版只需要：`buildContainer('cloud')`，业务层零修改。

## 4. LLM 多厂商兼容设计（核心需求）

### 4.1 抽象接口

```ts
// src/domain/ports/ILLMProvider.ts
export interface ILLMProvider {
  readonly id: string;
  readonly capabilities: {
    streaming: boolean;
    jsonMode: boolean;
    maxContextTokens: number;
    supportsSystemPrompt: boolean;
  };

  chat(req: ChatRequest): Promise<ChatResponse>;
  chatStream?(req: ChatRequest): AsyncIterable<ChatChunk>;
}

export interface ChatRequest {
  model: string;
  system?: string;
  messages: Array<{ role: 'user' | 'assistant'; content: string }>;
  temperature?: number;
  maxTokens?: number;
  jsonSchema?: object;        // 结构化输出（断句任务必备）
  metadata?: Record<string, unknown>;
}
```

### 4.2 四种适配器

| 适配器 | 兼容厂商 | 配置项 |
|---|---|---|
| **OpenAICompatibleProvider** | OpenAI、DeepSeek、Moonshot、智谱、OpenRouter、Together、Groq、Ollama、vLLM、LM Studio、任何兼容 `/v1/chat/completions` 的服务 | `baseURL`、`apiKey`、`model`、可选 `headers` |
| **AnthropicProvider** | Claude 官方 / 兼容 Anthropic Messages API 的代理 | `baseURL`、`apiKey`、`anthropic-version` |
| **GeminiProvider** | Google AI Studio / Vertex AI（公开端点） | `apiKey`、`model` |
| **CustomTemplateProvider** | 任意 HTTP API | `baseURL`、`headers`、`requestTemplate`、`responseExtractor` |

### 4.3 用户配置数据结构

```ts
type ProviderConfig =
  | { type: 'openai-compatible'; id: string; name: string; baseURL: string;
      apiKey: string; model: string; headers?: Record<string,string> }
  | { type: 'anthropic'; id: string; name: string; baseURL?: string;
      apiKey: string; model: string }
  | { type: 'gemini'; id: string; name: string; apiKey: string; model: string }
  | { type: 'custom'; id: string; name: string; baseURL: string; method: 'POST';
      headers: Record<string,string>;
      requestTemplate: string;     // 含 {{system}} {{messages}} 占位符
      responseExtractor: string }; // JSONPath，例 $.choices[0].message.content

interface LLMSettings {
  providers: ProviderConfig[];
  activeProviderId: string;
  // 不同任务可绑定不同 provider（自用阶段先全局一个，留好扩展点）
  taskBindings?: { translation?: string; segmentation?: string; vocab?: string };
}
```

### 4.4 自定义模板示例（兼容任意奇葩 API）

```yaml
# 用户在 Options 页填写
type: custom
baseURL: https://my-proxy.com/api/llm
method: POST
headers:
  Authorization: Bearer xxx
  Content-Type: application/json
requestTemplate: |
  {
    "input": "{{system}}\n\n{{lastUserMessage}}",
    "params": {"temp": 0.3}
  }
responseExtractor: "$.data.output_text"
```

### 4.5 Provider Registry & 故障降级

- `ProviderRegistry` 负责按 `activeProviderId` 取实例
- 后期可扩展：失败自动降级到备用 provider、不同任务绑定不同 provider（断句用便宜模型，翻译用高质量模型）

## 5. MVP 范围切分

### 5.1 必须做（M1–M5）

| 编号 | 功能 | 验收 |
|---|---|---|
| F1 | YouTube 页面注入 + 检测视频 ID 切换 | 切视频 / 翻页都能正确触发 |
| F2 | 抓取 YouTube 原生字幕（timedtext API） | 原文准确，时间戳对齐 |
| F3 | 智能断句（LLM 句界检测 + 重组） | 跨 cue 半句被正确合并 |
| F4 | 批量翻译（5–10 句一批，注入视频元信息） | 译文连贯不出戏 |
| F5 | 时间戳按字符比例重分配 | 中文字幕跟着英文播放节奏走 |
| F6 | 双语字幕渲染（覆盖在原生字幕容器上） | 中英对照、可开关 |
| F7 | Options 页：LLM provider 配置（CRUD） | 可加多 provider，切换生效 |
| F8 | Options 页：基础字幕样式（字号、颜色、位置） | 实时预览 |
| F9 | 字幕结果缓存（IndexedDB） | 二次打开秒开 |
| F10 | 侧边栏：完整剧本 + 点击跳转 | 点击精确定位到对应时间 |
| F11 | **逐词高亮（估算版）**：当前词下划线，跟随播放进度 | 在原生字幕场景下不漂移超过 ±200ms |
| F12 | **专业级双语默认排版**（见第 15 节） | 不调任何设置就能在多种背景视频下清晰可读 |

### 5.2 不在 MVP（先留接口）

- ASR 转录、**词级 forced alignment（高精度逐词高亮）**、划词查词、深度样式定制、跨设备同步、Pro 功能门控

## 8. 核心数据流

### 8.1 主链路：打开视频 → 显示双语字幕

```
[YouTube 页]
  │ (DOMBinding 检测 videoId 变化)
  ▼
[Content Script] ── postMessage ──▶ [Background SW]
                                       │
                                       ▼
                          SubtitlePipelineUseCase.run(videoId)
                                       │
                       ┌───────────────┼───────────────┐
                       ▼               ▼               ▼
                ICache.get()    ISubtitleSource    ILLMProvider
                (命中?)          .fetch(videoId)    .chat(...)
                       │               │               │
                       ▼               ▼               ▼
                  返回缓存      原始 cues[]    断句结果 + 译文
                       │               │               │
                       └───────┬───────┘               │
                               ▼                       │
                  SmartSegmentationUseCase  ◀──────────┘
                               │
                               ▼
                  TranslateBatchUseCase
                               │
                               ▼
                  TimestampAllocator (重新分配时间)
                               │
                               ▼
                       ICache.put()
                               │
                               ▼
              ◀── postMessage（带分段译文）── [Background]
                               │
                               ▼
                   [Content Script: SubtitleOverlay]
                               │
                               ▼
                       渲染到 Shadow DOM
```

### 8.2 智能断句算法（核心）

**输入**：原始 cues `[{ start: 12.3, end: 14.1, text: "And that's why we" }, { start: 14.1, end: 16.0, text: "decided to rewrite the entire" }, ...]`

**步骤**：

1. **预处理**：合并相邻 cue（保留每段对应的原始 cue 索引数组）
2. **句界检测**：
   - 优先用启发式（句末标点 `.?!`、大写开头）粗切
   - 跨 cue 模糊处的让 LLM 判断（结构化 JSON 输出 `{ boundaries: [3, 7, 12, ...] }`）
   - 一次喂 ~20 个 cue，节省调用次数
3. **句子重组**：根据边界拼成 `Sentence[]`，每个 sentence 关联其覆盖的 cue 索引区间和原始时间区间
4. **批量翻译**：
   - 一批 5–10 句送 LLM，要求输出 JSON 数组（保证可解析）
   - System prompt 注入：视频标题、章节信息、频道、风格选项
   - 上一批最后一句的译文作为下一批 context（连贯性）
5. **时间戳重分配**：每个 sentence 内部按中文字符长度等比划分回原始 cue 区间，得到新的 `BilingualCue[]`
6. **写入缓存**：key = `${videoId}:${modelId}:${targetLang}:${styleVersion}`

**Prompt 范式（断句）**：
```
你是字幕断句专家。给你一组 YouTube 字幕片段，每段以 [n] 编号。
请输出哪些位置是完整句子的结束（换行不算），返回 JSON。
视频标题：{{title}}

[1] And that's why we
[2] decided to rewrite the entire
[3] codebase last quarter.
[4] But that's not the whole story.
...

输出格式：{ "boundaries": [3, 4, ...] }   // 这些编号的 cue 末尾是句尾
```

**Prompt 范式（翻译）**：
```
你是专业字幕翻译。视频背景：
- 标题：{{title}}
- 频道：{{channel}}
- 章节：{{currentChapter}}
- 风格：{{styleHint}}   // "信达雅" / "直白白话" / "技术术语保留英文"
上一句已译为：{{prevTranslation}}（仅供风格连贯参考）

请翻译以下英文句子为中文，保持原意不增删，输出 JSON 数组：
[
  "Sentence 1...",
  "Sentence 2...",
  ...
]
输出：{ "translations": ["...", "..."] }
```

### 8.3 词级时间戳获取（逐词高亮的基石）

| 策略 | 精度 | 成本 | 适用阶段 |
|---|---|---|---|
| **A. 字符等比估算** | ±200–400ms（口语停顿处可能漂移） | 0 | MVP（F11）|
| **B. Forced Alignment** (WhisperX / wav2vec2) | ±50ms | 拉音轨 + 推理（GPU 友好） | Phase 3 升级 |
| **C. ASR with `word_timestamps`** | ±50ms | 整段重新 ASR | 仅在原生字幕缺失时用（合并进 ASR 路径）|

**词切分规则**（关键，雪球细节）：
- 英文：以空格切分，**标点紧跟前一个词**（`documents,` 作为一个高亮单元，不单独高亮逗号）
- 缩写整体保留：`don't` / `it's` 不切
- 数字/连字符词整体保留：`2026-04-27` / `state-of-the-art`
- 中文：MVP 不做逐词高亮（粒度模糊），整句轻微淡入即可

**估算算法（MVP）**：
```
对每个 cue：
  total_chars = sum(len(word) + 1 for word in words)  // +1 for trailing space/punct
  cumulative = 0
  for word in words:
    word_start = cue.start + (cumulative / total_chars) * (cue.end - cue.start)
    word_end   = cue.start + ((cumulative + len(word)) / total_chars) * (cue.end - cue.start)
    cumulative += len(word) + 1
```
后续可用 **VAD（静音检测）** 做局部修正：拿到原始音频用 ffmpeg.wasm 切 30s 片段做 VAD，把"明显停顿"作为词边界对齐参考。

### 8.4 关键性能策略

- **首屏优先**：先翻译当前播放位置 ±60s 的 cue，其余在后台流式补齐
- **流式渲染**：LLM 流式输出时，每收到一句就立即更新 UI（用户感知"6 秒内出字幕"）
- **二级缓存**：内存（当前会话）+ IndexedDB（跨会话）
- **断点续传**：缓存粒度到「批次」，部分失败时只重试失败批

## 9. 商业化冗余设计

以下能力在 MVP 中以**接口/空实现**形式预留，商业化阶段直接替换实现：

| 维度 | MVP 实现 | 商业化实现 | 切换方式 |
|---|---|---|---|
| **LLM 调用** | 用户 key 直连厂商 | 后端代理（后端持 key + 计费） | 增加一个 `RemoteProxyProvider`，用户配置切到此即可 |
| **认证授权** | `LocalAuthGate`（永远放行） | `RemoteAuthGate`（JWT + 服务端校验） | Container 装配换实现 |
| **额度门控** | `IAuthGate.isAllowed()` 永真 | 调远端 `/quota` 接口 | 同上 |
| **遥测** | `NoopTelemetry` | `RemoteTelemetry` (PostHog / 自建) | 同上 |
| **配置同步** | 仅本地 `chrome.storage` | 增加 `IRemoteSync` 接口 + 实现 | 新增 Port，不影响业务层 |
| **字幕共享库** | 仅本地缓存 | 增加 `ISharedSubtitleRepo` Port | 新增 Port |
| **错误上报** | console + 本地日志 | Sentry / 自建 | 适配 `ITelemetry` |

**核心原则**：业务层（`domain/`）一行不改的前提下，只通过 `composition/container.ts` 的不同装配，从 MVP 切到云版。

## 11. 关键风险与应对

| 风险 | 等级 | 应对 |
|---|---|---|
| YouTube timedtext API 反爬/变更 | 高 | 多入口（player config / page HTML scrape）+ 失败日志监控 |
| LLM 调用 JSON 输出不稳定 | 中 | 强制 `response_format: json_object` / 结构化输出；解析失败重试一次降级到非 JSON 模式 |
| MV3 service worker 30s 死亡 | 中 | 长任务拆批 + offscreen document 备用 |
| 断句错误导致时间戳错位 | 中 | 算法层加 sanity check（相邻句时间不允许重叠/反向）+ 出错回退到原 cue |
| 自定义模板 provider 安全（用户写错把 key 发错地方） | 中 | URL 白名单提示、首次调用前 ping、密钥不允许进 requestTemplate 占位符 |
| 不同厂商 token 计费/限流差异大 | 中 | 抽象 `RateLimiter` + 每 provider 独立配额 |
| 长视频成本失控（自用） | 低 | UI 显示预估 token 数 + 用户确认才开跑 |
| 商业化时合规（YouTube TOS、用户数据） | 高 | 提前看 TOS；用户数据加密存储；后端不留视频原文超过缓存周期 |

## 15. 双语字幕排版规范（默认即专业级）

> 对标雪球：「不调样式就好看」是用户留存的关键。本节规定 MVP 出厂默认值。

### 15.1 视觉层级

```
┌─────────────────────────────────────────────────┐
│                                                 │ ← 视频画面
│                                                 │
│      一个演示文稿、电子表格、文档              │ ← 中文（主要信息）
│                                                 │
│   A deck, spreadsheets, documents, the         │ ← 英文（次要信息 + 当前词高亮）
│              ‾‾‾‾‾‾‾‾‾‾                         │
│                                                 │
└─────────────────────────────────────────────────┘
        ↑ 距底部 12% 视频高度
```

### 15.2 字体与字号

| 角色 | 字体（CSS fallback 链） | 字重 | 字号 | 颜色 |
|---|---|---|---|---|
| 中文译文（主） | `"PingFang SC", "Microsoft YaHei UI", "Noto Sans SC", system-ui, sans-serif` | **800** (Heavy) | 视频高度 × **3.4%** | `#FFFFFF` |
| 英文原文（次） | `"Inter", "SF Pro Text", "Segoe UI", system-ui, sans-serif` | 500 (Medium) | 中文字号 × **0.72** | `#C8D0DC` |
| 当前词（高亮） | 同英文 | 500 | 同英文 | 同英文 + 下划线 `#5BA3FF` 2px |

字号绑定视频高度（用 `ResizeObserver` 监听播放器尺寸）：全屏自动放大、画中画自动缩小。

### 15.3 描边 / 阴影（关键：不要叠加背景框）

雪球的舒服感来自「**只用描边和阴影抗背景，不放半透明黑底**」——画面更干净。

```css
.cue-zh {
  -webkit-text-stroke: 1px rgba(0,0,0,0.85);
  text-shadow:
    0 1px 2px rgba(0,0,0,0.55),
    0 0 6px rgba(0,0,0,0.35);   /* 双层阴影：贴边 + 漫射 */
  paint-order: stroke fill;     /* 描边在下，字面在上，避免锯齿 */
}
.cue-en {
  text-shadow: 0 1px 2px rgba(0,0,0,0.6);
}
```

提供「视频太亮时自动加薄背景」的 fallback：检测当前播放帧主色亮度（采样 5×5 块），亮度 > 0.75 时启用 `backdrop-filter: blur(8px)` + 半透明 `rgba(0,0,0,0.25)`。

### 15.4 当前词高亮规则

- **形式**：仅 **2px 蓝色下划线**，不变色、不加粗、不放大（避免布局抖动）
- **颜色**：`#5BA3FF`（亮蓝，与白/灰主体形成色相对比但不刺眼）
- **过渡**：词切换时 `transition: all 80ms ease-out`，不用动画，太快太慢都不舒服
- **单元**：词 + 紧邻标点作为一个整体单元（`documents,` 一起高亮）
- **行尾边界**：当前词跨行时下划线截断到行尾，**不**在两行间画连接线

### 15.5 间距

| 项 | 值 |
|---|---|
| 中英行间距 | 中文行高 × 0.55 |
| 中文行高 | `1.32` |
| 英文行高 | `1.42` |
| 距视频底部 | 视频高度 × 12%（避开 YouTube 进度条悬浮区） |
| 字间距 (letter-spacing) | 中文 0.02em / 英文 0.01em |

### 15.6 双行 / 三行处理

- 单 cue 渲染为 1 行中文 + 1 行英文，**不主动换行**
- 句子过长溢出时按词边界换行，最多 2 行（再多说明断句失败，要触发回退）
- 中英对齐方式：均**居中**（左对齐在中英混排时视觉重心会偏）

### 15.7 用户可改 / 不可改

| 项 | MVP 可改 | 备注 |
|---|---|---|
| 字号缩放（80%–140%） | ✅ |  |
| 中文/英文颜色 | ✅ |  |
| 字体（系统字体下拉） | ✅ |  |
| 距底部位置 | ✅ |  |
| 描边样式 | ❌ | Phase 3 开放 |
| 高亮颜色 | ❌ | Phase 3 开放 |
| 排版结构（中上英下） | ❌ | 永不开放，是设计决策 |

### 15.8 Shadow DOM 隔离

字幕容器挂载在 Shadow DOM 内（`mode: 'closed'`），所有样式作用域内，**不受 YouTube 全局 CSS 干扰**，YouTube 改版时受影响最小。
